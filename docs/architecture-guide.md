## 1. nicoアプリって何？

**nico** は「優しいSNS」です。

普通のSNSでは批判や攻撃的なコメントが飛び交うことがありますが、
nicoでは **共感・賞賛のコメントだけ** が投稿できます。

### 主な機能

| 機能 | 説明 |
|------|------|
| 投稿 | 自分の気持ちやできごとを匿名で投稿できる |
| リアクション | 「わかる」「すごい」「えらい」の3種類で反応できる |
| コメント | AIが「共感・賞賛か？」を判定して、優しいコメントだけ通す |
| LINE共有 | 友達にnicoを紹介できる |

│  │(ブラウザ) │──→│(サーバー) │──→│(Firestore DB)│     │
│  │          │   │          │   │              │     │
│  │ 画面表示  │   │ AI判定    │   │  データ保存   │     │

│                                                       │
│  ┌──────────────┐                                    │
│  │  外部シェフ    │                                    │

│  └──────────────┘                                    │
└─────────────────────────────────────────────────────┘
```

| レストランの例 | nicoアプリでの意味 |

| 外部シェフ | OpenAI API。コメントが「優しいか」を味見してくれる専門家 |

## 5. 画面の構成（ページ一覧）

nicoには **3つの画面** があります。

### 画面①：トップページ（タイムライン）
**ファイル:** `app/page.tsx`
**URL:** `/`

```
┌──────────────────────────┐
│  🟢 nico    [LINE] [投稿] │  ← Header
│  ID: a1b2...z9y8         │
├──────────────────────────┤
│ ┌────────────────────┐   │
│ │ 匿名ユーザー (abc..)│   │  ← PostCard ×n件
│ │ 今日ちょっと頑張った  │   │
│ │ [わかる3][すごい1]   │   │  ← ReactionButtons
│ │ 💬 3    詳細を見る→  │   │
│ └────────────────────┘   │
│ ┌────────────────────┐   │
│ │ 次の投稿...          │   │
│ └────────────────────┘   │
└──────────────────────────┘
```

**やっていること:**
1. 画面が開かれたら Firestore から投稿一覧を取得
2. 新しい順に PostCard を並べて表示
3. リアクションを押すと即座にカウント更新

### 画面②：新規投稿ページ
**ファイル:** `app/post/new/page.tsx`
**URL:** `/post/new`

```
┌──────────────────────────┐
│  入力してね               │
│ ┌────────────────────┐   │
│ │                    │   │  ← テキスト入力欄
│ │   (140文字まで)     │   │
│ └────────────────────┘   │
│  0/140文字               │
│ [戻る]         [投稿する] │
└──────────────────────────┘
```

**やっていること:**
1. テキスト入力（1〜140文字）
2. バリデーション（空チェック・文字数チェック）
3. レート制限（1分に1回まで）
4. Firestoreに保存 → トップに戻る

### 画面③：投稿詳細ページ
**ファイル:** `app/post/[id]/page.tsx` + `PostDetailClient.tsx`
**URL:** `/post/abc123`（abc123 = 投稿ID）

```

│  🟢 nico    [LINE] [投稿] │
│  ← トップに戻る           │

## 6. コンポーネント構成（UIの部品）

### レゴブロックのようなもの

コンポーネントとは **レゴブロック** のようなものです。
小さなブロック（部品）を組み合わせて、大きな完成品（ページ）を作ります。

```
トップページ (page.tsx)
 ├── Header（ヘッダー）
 │    └── Logo（ロゴ画像）
 │
 └── PostCard（投稿カード）× n件
      └── ReactionButtons（リアクションボタン）

詳細ページ (PostDetailClient.tsx)
 ├── Header
 │    └── Logo
 ├── PostCard（詳細モード）
 │    └── ReactionButtons
 ├── CommentList（コメント一覧）
 └── CommentInput（コメント入力）
```

### 各コンポーネントの役割

| コンポーネント | レストランの例 | 役割 |
|---|---|---|
| **Header** | お店の看板＋入口 | ロゴ表示、LINE共有ボタン、投稿ボタン |
| **Logo** | お店のロゴマーク | ロゴ画像を表示するだけ |
| **PostCard** | 料理プレート | 1件の投稿の見た目をまとめて表示 |
| **ReactionButtons** | 「美味しい」ボタン | 「わかる」「すごい」「えらい」を押せる |
| **CommentList** | レビュー一覧 | その投稿へのコメントを一覧表示 |
| **CommentInput** | レビュー投稿フォーム | コメントを入力して送信する |

### props（プロップス）= 部品への指示書

コンポーネントに渡すデータを **props** と言います。

```
日常の例：
  ピザ屋で注文するとき →「マルゲリータ、Lサイズ、チーズ増量」と伝える
  
コードの例：
  <PostCard post={投稿データ} showDetailLink={true} />
             ↑ どの投稿？      ↑ 詳細リンクを出す？
```

## 7. データの流れ（注文から料理が届くまで）

### 投稿一覧を表示する流れ

```
① ユーザーがトップページを開く
      ↓
② page.tsx の useEffect が動く
   「投稿一覧ください！」
      ↓
③ fetchPosts() が Firestore に問い合わせ
   「posts コレクションを新しい順でちょうだい」
      ↓
④ Firestore が投稿データを返す
   [{id:"abc", content:"頑張った", ...}, ...]
      ↓
⑤ state (posts) にデータがセットされる
      ↓
⑥ React が画面を再描画（PostCard が並ぶ）
      ↓
⑦ ユーザーに投稿一覧が見える！
```

**日常の例:**
1. レストランに入る（ページを開く）
2. 「今日のおすすめ一覧ください」（データ取得リクエスト）
3. キッチンが冷蔵庫を確認（Firestoreに問い合わせ）
4. 食材リストを返す（データが返ってくる）
5. メニュー表に書く（stateにセット）
6. テーブルにメニューを置く（画面を描画）
7. お客さんがメニューを見る（ユーザーが見る）

### コメント送信の流れ（AI判定あり）

```
① ユーザーが「わかる！頑張ったね」と入力して送信を押す
      ↓
② クライアント側バリデーション
   ・空じゃない？ → OK
   ・100文字以内？ → OK
   ・NGワード含む？ → OK（含まない）
   ・30秒経った？ → OK
      ↓
③ /api/judge-comment にリクエスト送信
   「投稿本文：今日頑張った、コメント：わかる！頑張ったね」
      ↓
④ サーバー側（route.ts）で処理
   ├── NGワードチェック → OK
   ├── 簡易パターン判定（「わかる」が含まれる→即YES）
   │   → YES ならここで OK を返す（AI呼ばない＝コスト節約）
   │
   └── パターンに該当しない場合
       ↓
       OpenAI API に問い合わせ
       「この投稿に対するこのコメントは共感的？」
       → YES / NO
      ↓
⑤ 判定結果 { ok: true } が返る
      ↓
⑥ Firestore に保存（transaction）
   ├── comments コレクションにコメント追加
   └── posts の commentCount を +1
      ↓
⑦ 画面を更新（コメント一覧＋コメント数を再取得）
```

**日常の例:**
```
① お客さんがレビューを書く
② 受付で「白紙じゃないか」「長すぎないか」をチェック
③ レビューをキッチンに渡す
④ キッチンで判定：
   ├── 明らかなNGワードは即却下
   ├── 「おいしい」「最高」→ 明らかにOK → そのまま掲示
   └── 微妙な表現 → 外部の専門家（AI）に味見を依頼
⑤ 専門家「これは褒め言葉ですね OK」
⑥ レビューボードに貼る（DB保存）
⑦ 他のお客さんにも見えるようにする（画面更新）
```

## 8. 匿名ユーザーの仕組み

### ログインしないSNS

nicoには **会員登録もログインもありません**。
代わりに **匿名ID** をブラウザが自動で発行します。

```
初回アクセスの流れ：

① nico を初めて開く
      ↓
② useAnonUserId フック が動く
      ↓
③ getOrCreateAnonUserId() を呼ぶ
      ↓
④ localStorage を確認
   「anonUserId」というキーで保存済み？
      ↓
   ├── YES → その ID を使う
   └── NO  → crypto.randomUUID() で新規ID生成
              → localStorage に保存（次回からこれを使う）
      ↓
⑤ ID を state にセット → 画面に「ID: a1b2...z9y8」と表示
```

**日常の例:**
- コンビニのポイントカードで例えると…
- 初回来店 → 新しいカードを発行してくれる（UUID生成）
- 次回来店 → 財布からカードを出す（localStorageから取得）
- カードの番号があなたの「匿名ID」
- 名前は書いてない（匿名）けど、同じ人かどうかはわかる

## 9. 投稿の仕組み

### Firestore のデータ構造

```
Firestore（データベース）
 └── posts（コレクション = フォルダ）
      ├── abc123（ドキュメント = 1件の投稿）
      │    ├── content: "今日ちょっと頑張った"
      │    ├── userId: "9f2a-...-c81d"
      │    ├── createdAt: 2026/04/13 10:30
      │    ├── reactionCounts:
      │    │    ├── wakaru: 3
      │    │    ├── sugoi: 1
      │    │    └── erai: 0
      │    └── commentCount: 2
      │
      ├── def456（別の投稿）
      │    └── ...
      └── ...
```

**日常の例:**
- Firestore = 図書館
- コレクション (posts) = 「小説」の棚
- ドキュメント (abc123) = 1冊の本
- フィールド (content, userId, ...) = 本の中身（タイトル、著者名、ページ数…）

### 投稿の保存処理 (`createPost`)

```typescript
// こういうデータを Firestore に保存する
await addDoc(colRef, {
  content: "今日頑張った",       // 投稿本文
  userId: "9f2a-...",           // 誰が書いたか
  createdAt: serverTimestamp(), // サーバーの時計で記録
  reactionCounts: { wakaru: 0, sugoi: 0, erai: 0 },  // 最初は全部0
  commentCount: 0,              // コメントも最初は0
});
```

## 10. リアクションの仕組み

### 3つのリアクション

| ボタン | 意味 | 使う場面 |
|--------|------|----------|
| わかる | 共感 | 「あー、その気持ちわかる」 |
| すごい | 称賛 | 「それはすごいね！」 |
| えらい | 努力承認 | 「頑張ったね、えらい！」 |

### トグル動作（押すたびに切り替わる）

```
パターンA: まだ何も押してない → 「わかる」を押す
  → reactions に追加 & posts.wakaru +1

パターンB: 「わかる」を押してある → もう一度「わかる」を押す
  → reactions を削除 & posts.wakaru -1（取り消し）

パターンC: 「わかる」を押してある → 「すごい」を押す
  → wakaru -1, sugoi +1（乗り換え）
```

**日常の例:**
- Amazonの「参考になった」ボタンと同じ
- 押すと+1、もう一度押すと取り消し
- 違うのは「3種類から選べる」こと（Amazonは1種類）

### トランザクション（整合性の保証）

```
「reactionカウントがおかしくならない」ための仕組み

例えば…悪い例（トランザクションなし）：
  ① Aさんが「わかる」を押す → wakaru: 3 → 4 にする
  ② 同時にBさんも「わかる」を押す → wakaru: 3 → 4 にする
  結果: 本当は 5 なのに 4 になる！

トランザクションあり：
  ① Aさんが更新中 → Bさんは待つ
  ② Aさん完了（wakaru: 4）
  ③ Bさんがやり直し → wakaru: 4 → 5 にする
  結果: 正しく 5 になる ✅
```

**日常の例:**
- 銀行のATMと同じ。
- 2人が同じ口座から同時に引き出そうとしたら、1人ずつ順番に処理する
- そうしないと残高がおかしくなる

## 11. コメントの仕組み（AI判定つき）

### 3段階の安全フィルター

nicoのコメントは **3つの関門** を通過しないと投稿できません。

```
コメント「わかる！頑張ったね」
      ↓
┌─────────────────────────────────┐
│ 第1関門：クライアント側チェック     │
│  ├── 空じゃない？ ✅              │
│  ├── 100文字以内？ ✅             │
│  ├── NGワード含む？ ✅（含まない） │
│  └── 30秒経った？ ✅              │
└────────────┬────────────────────┘
             ↓
┌─────────────────────────────────┐
│ 第2関門：サーバー側チェック         │
│  ├── 文字数OK？ ✅                │
│  ├── NGワード？ ✅                │
│  └── 共感パターン？ → 「わかる」   │
│      含む → 即YES ✅              │
│     （AI呼ばずにコスト節約）       │
└────────────┬────────────────────┘
             ↓
┌─────────────────────────────────┐
│ 第3関門：AI判定（微妙な場合のみ）   │
│  OpenAI に質問：                  │
│  「この投稿に対するこのコメントは   │
│   共感的ですか？」                 │
│  → YES / NO                      │
└────────────┬────────────────────┘
             ↓
          保存 or 拒否
```

**日常の例:**
```
テーマパークの入場に例えると：

第1関門（入口ゲート）：チケット持ってる？ 身長OK？
  → ここで明らかにダメな人は入れない

第2関門（手荷物検査）：危ないもの持ってない？
  → 明らかに安全なら通す（水筒だけの人は即OK）

第3関門（VIPエリア審査）：本当に招待状の主か？
  → 微妙な場合だけ専門スタッフが確認
```

### NGワードフィルター

```
即ブロックされるワード（例）：
  死ね, 殺す, バカ, ゴミ, キモい, うざい, ...

ブロックされないワード（文脈次第）：
  嫌だ, 最悪, つらい
  → これらは「嫌だよね、わかる」のように
    共感で使われることがあるので、AIに判断を委ねる
```

### AI判定の仕組み

```
OpenAI に渡すメッセージ：

System（AIへの指示）:
  「あなたはSNSのモデレーターです。
   投稿(POST)に対するコメント(COMMENT)が
   共感的かどうかを YES/NO だけで答えて」

User（実際のデータ）:
  「POST: 同僚が上司に媚びてる
   COMMENT: 嫌だよねそういうの。わかる。」
   → YES（共感）

  「POST: 今日mtg10件もやった！
   COMMENT: 嫌だよねそういうの。」
   → NO（ポジティブな投稿に対してネガティブ = 水差し）
```

## 12. Server Component と Client Component

### Next.js での「2種類のコンポーネント」

```
Server Component（サーバー側で動く）
  = レストランのキッチン
  → お客さんには見えない場所で準備する
  → 完成した料理（HTML）をテーブルに運ぶ

Client Component（ブラウザ側で動く）
  = テーブルの上の卓上コンロ
  → お客さんが自分で操作できる（ボタン押す、入力する）
```

| 種類 | 特徴 | 例 |
|------|------|------|
| Server Component | ボタンやイベントは使えない。データ取得向き | `app/post/[id]/page.tsx` |
| Client Component | `"use client"` を書く。ボタンやstateが使える | `Header.tsx`, `PostCard.tsx` |

### なぜ分けるの？

```
Server Component のメリット：
  ✅ APIキーなど秘密情報を安全に扱える
  ✅ ページの読み込みが速い（HTMLが完成した状態で届く）
  ✅ ブラウザに送るJavaScriptが少なくて済む

Client Component のメリット：
  ✅ ボタンクリック、入力、アニメーションができる
  ✅ useState, useEffect などReactの機能が使える
  ✅ ブラウザのAPI（localStorage, window）にアクセスできる
```

### nicoアプリでの使い分け

```
app/post/[id]/page.tsx        ← Server Component
  └── PostDetailClient.tsx    ← Client Component（"use client"）

なぜ分けている？
  page.tsx（サーバー側）:
    → URL の [id] パラメータを安全に取り出す（await params）
    → それを PostDetailClient に渡す

  PostDetailClient.tsx（ブラウザ側）:
    → ボタン操作、状態管理、Firestore通信など
    → ユーザーが実際に触る部分
```

## 13. セキュリティと安全設計

### なぜ気にするの？

SNSは誰でもアクセスでき、悪意あるユーザーもいます。
nicoでは以下の対策をしています。

| 対策 | 何を守る | 日常の例 |
|------|----------|----------|
| APIキーをサーバー側に置く | OpenAIキーの漏洩防止 | 金庫の鍵を客席に置かない |
| レート制限 | 連続投稿/連続コメントの防止 | 「お1人様1個限り」 |
| NGワードフィルター | 暴言の即時ブロック | 入口の手荷物検査 |
| AI判定 | 巧妙な攻撃コメントの検知 | 専門家による身元確認 |
| トランザクション | データの不整合防止 | ATMの順番待ち処理 |
| `"use client"` の分離 | サーバー秘密情報の保護 | キッチンとホールの仕切り |
| 入力バリデーション（両側） | 不正データの防止 | 表と裏の両方でチェック |

### 重要：「クライアント側」と「サーバー側」両方でチェック

```
なぜ両方でチェックする？

クライアント側（ブラウザ）だけだと…
  → 開発者ツールで JavaScript を書き換えられる
  → フィルターをすり抜けられてしまう

サーバー側（API）だけだと…
  → ユーザーが入力するたびにサーバーに聞きに行く
  → 待ち時間が長くてストレスになる

だから両方でやる！
  クライアント側 = UXのため（すぐフィードバック）
  サーバー側 = セキュリティのため（すり抜け防止）
```
